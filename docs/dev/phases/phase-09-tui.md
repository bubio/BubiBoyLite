# フェーズ 9: TUI・最近使ったファイル

## 前提

- 依存フェーズ: 5（ゲームが動くこと）。8 と並行可能（recent の永続化は T8-1 の config 機構を使うので T8-1 完了後が楽）。
- BluePrint: 「Claude Code などに見られる TUI も提供する」「--recent : 最近使ったファイルを表示して選択できる。ROM ファイルの指定がある場合は、そちらを優先」

## ゴール

`bbl` を引数なしで起動すると TUI が立ち上がり、ROM 選択 → ゲーム起動 → 終了までを完結できる。
`--recent` で履歴からの選択ができる。

## フェーズ完了の検証コマンド

```sh
./bbl              # TUI 起動 → ROM 選択 → プレイ → 戻る → q（一巡、目視）
./bbl --recent     # 履歴から選択できる
./bbl --recent <rom>   # ROM 指定が優先される
odin test tests -collection:bbl=src   # recent 履歴の単体テスト PASS
```

## TUI の設計方針（実装前に読むこと）

- 依存ライブラリなし。ANSI エスケープシーケンス + ターミナル raw mode を自前で扱う（`src/app/tui.odin` に閉じる）
- 画面構成（Claude Code 風 = 枠付きパネル + ステータスライン + キーヒント常時表示）:
  ```
  ┌─ BubiBoyLite v0.1.0 ──────────────────────┐
  │  ROM を選択してください                     │
  │  ▸ ~/roms/game1.gbc          (MBC5, CGB)  │
  │    ~/roms/game2.gb           (MBC1)       │
  │    [ディレクトリを開く…]                    │
  └───────────────────────────────────────────┘
   ↑↓ 選択  Enter 起動  q 終了
  ```
- ゲーム実行中は SDL ウィンドウが主役。TUI は起動元ターミナルにステータス行（FPS / 音量 / スロット）を出し続ける
- 端末が TTY でない（パイプ等）場合は TUI を起動せずエラーメッセージ + exit 1

### T9-1: ターミナル制御基盤

- [ ] 完了

**目的**: raw mode・ANSI 描画・リサイズ対応の TUI 基盤を作る。
**作るもの**: `src/app/tui.odin`:
- raw mode: POSIX は termios（`ECHO|ICANON` off、`VMIN=0/VTIME=1`）、Windows は `SetConsoleMode`（`ENABLE_VIRTUAL_TERMINAL_PROCESSING` 有効化含む）
- 代替スクリーンバッファ (`\x1b[?1049h/l`)、カーソル非表示、終了時の確実な復元（defer + シグナル/パニック時も）
- 描画: 全画面を文字列バッファに構築して一括 write（ちらつき防止）
- キー入力: 矢印（CSI シーケンス）、Enter、Esc、q の読み取り
- 端末サイズ取得とリサイズ追従
**参照**: TUI 設計方針（上記）。Odin の `core:sys/posix`（termios）/ `core:sys/windows`
**完了条件 (DoD)**: デモ画面（枠 + カーソル移動）が表示され、q で元のターミナル状態に完全復元される。`bbl --headless` 等の非 TUI 経路に影響なし。
**検証方法**:
```sh
./bbl               # TUI デモが出て q で復元
./bbl -v            # TUI を経由せず従来どおり
echo | ./bbl        # 非 TTY: エラーメッセージ + exit 1
```
**落とし穴**: 復元漏れするとユーザーのターミナルが壊れる。異常終了パスでも必ず復元する（Odin の defer はパニックでも走るが、os.exit は defer を飛ばすので TUI 起動中の exit は復元関数経由にする）。Windows 10 の古いコンソールでは VT 処理の有効化が必須。
**依存**: なし

### T9-2: ROM 選択画面

- [ ] 完了

**目的**: TUI のメイン画面。ディレクトリの ROM 一覧から選択して起動する。
**作るもの**: tui.odin:
- 起動ディレクトリ（設定 `rom_dir` があればそこ）の `.gb/.gbc` を列挙、ヘッダを読んで MBC 種別/CGB フラグを右側に表示（T4-1 の cartridge.odin を再利用）
- ↑↓ 選択、Enter で起動、ディレクトリ移動（`..` 含む）、q 終了
- 選択して Enter → TUI を**一時停止**（画面復元）→ SDL でゲーム実行 → ゲーム終了後 TUI に戻る
**参照**: BluePrint.md「特徴」、`src/core/cartridge.odin`（T4-1）
**完了条件 (DoD)**: `bbl` 引数なしで TUI → ROM 選択 → プレイ → ウィンドウクローズ → TUI に戻る → q 終了、が一巡する。
**検証方法**: 上記フローを目視で一巡。
**落とし穴**: ROM 一覧のヘッダ読みはファイル先頭 0x150 バイトだけ読む（巨大ディレクトリでの全読み込み禁止）。
**依存**: T9-1

### T9-3: 最近使ったファイル履歴と --recent

- [ ] 完了

**目的**: BluePrint の `--recent` を実装する。
**作るもの**: `src/app/recent.odin`:
- 履歴ファイル: 設定ファイルと同じ場所に `bbl_recent.txt`（1 行 1 パス、最新が先頭、最大 20 件、重複は先頭へ移動）
- ROM 起動成功のたびに更新。存在しなくなったパスは表示時にスキップ
- `--recent`: TUI の ROM 選択画面を履歴リストで開く。**ROM 引数があれば --recent は無視してそちらを起動**（BluePrint 仕様）
- TUI 通常起動時も「最近使ったファイル」セクションを一覧の先頭に表示
**参照**: BluePrint.md「--recent」
**完了条件 (DoD)**: 単体テスト: 履歴の追加/重複/上限。結合: `./bbl --recent` で履歴選択、`./bbl --recent rom.gb` は rom.gb を直接起動。
**検証方法**:
```sh
odin test tests -collection:bbl=src
./bbl --recent
./bbl --recent <rom>   # ROM 優先
```
**落とし穴**: 履歴のパスは絶対パスで保存。**実ユーザー名を含むパスをログや画面に出すのは可（ユーザー自身の端末）だが、リポジトリのテストフィクスチャに残さない**。
**依存**: T9-1（一覧 UI）, T8-1（設定場所の解決を共有）

### T9-4: 実行中ステータス表示

- [ ] 完了

**目的**: ゲーム実行中、起動元ターミナルに有用な情報を出す（BluePrint「TUI を活かすアイデア」）。
**作るもの**: tui.odin / main.odin:
- 1 行ステータス（1 秒毎更新、`\r` 上書き）: `▶ game.gbc | 59.7 fps | vol 80% | slot 1 | MBC5+RAM | 双速`
- fps は直近 1 秒の実フレーム数。オーディオアンダーラン発生時は警告色で表示
- TUI 経由でない起動（`bbl rom.gb`）でも TTY なら表示（非 TTY なら出さない）
**参照**: BluePrint.md「TUIを活かすアイデアがあると助かります」
**完了条件 (DoD)**: 実行中に fps・音量・スロットが実際の状態と連動して表示される（目視）。
**検証方法**: `./bbl <ゲーム>` でステータス行を確認、`./bbl <ゲーム> > /dev/null` で出ないこと。
**落とし穴**: stdout はシリアル出力等のログと衝突しうるので、ステータス行は stderr に出す。
**依存**: T9-1

### T9-5: TUI からのステート・音量操作

**依存**: T9-1, T9-4, （ステート操作は T7-4）

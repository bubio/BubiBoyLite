# フェーズ 16: Now Playingパネル簡素化 + ヒント文言修正 + 最下部固定バグ調査

## 前提

- 依存フェーズ: 15(操作の完全スラッシュコマンド化+設定即時反映、完了済み)。
- 経緯: ユーザーが実機(macOS Terminal.app、119x41)でスクリーンショットを提示し、
  以下2点を指摘(2026-07-19):
  1. Now Playing パネル(ROM名/状態/fps/音量/スロット、`shell_content_now_playing`
     tui.odin:783-809)は不要 — 「実行状態はステータス表示で十分です」
  2. コマンドエリア(区切り線+ステータス行+入力行)がターミナル最下部ではなく
     中途半端な位置に表示されている

事前調査で判明した事実:

- **ステータス行(`status_line_format`、tui.odin:1696-1716)は既に ROM名・fps・音量・
  スロット・cart_label・速度・警告・直近メッセージを全て含んでいる**。Now Playing
  パネルの ROM/状態/fps/音量/スロット行(tui.odin:787-793)は完全に重複表示。
  ユーザーの指摘どおり不要
- Now Playing 画面のヒント文言(tui.odin:862)
  `"+/- 音量  1-4 スロット  s/l 保存/復元  p 一時停止  / コマンド"` は
  **フェーズ15(T15-1)で該当ホットキーを廃止した際に消し忘れたバグ**。存在しない操作を
  案内している
- 最下部固定バグについて: `tui_render_shell`(tui.odin:210-260)は `rows`(引数)から
  `content_rows = rows - SHELL_RESERVED_ROWS` を計算し、ちょうど `rows` 行を書く設計に
  なっており、渡された `rows` が正しければ理論上は必ず最下行に入力行が来る。
  `game_shell_draw`(tui.odin:817-868)は毎フレーム `tui_term_size()` を呼び直している
  ため、通常の取得失敗・古い値のキャッシュは考えにくい。**有力な仮説**: alt screen 進入
  (`ALT_SCREEN_ENTER :: "\x1b[?1049h"`、tui.odin:39、進入箇所は tui.odin:403,478)時に
  **DECSTBM スクロール領域のリセット(`\x1b[r`)を一切行っていない**。直前に動作していた
  別プログラム(ターミナルマルチプレクサ、他のTUIアプリ等)が独自のスクロール領域を
  設定したまま解除せずに終了していた場合、alt screen 自体は新しいバッファに切り替わっても
  スクロール領域はターミナル側のモードとして引き継がれる実装が一般的(xterm系の一般的
  挙動)。その場合、本来 rows=41 のつもりで改行を重ねても、古いスクロール領域
  (例: 1-29行目)の下端でスクロールが起き、コマンドエリアがその範囲内に押し留められる
  見た目になり得る。これはスクリーンショットの「約27〜29行目あたりで区切り線以降が
  止まっている」という見え方と整合する

## 対応方針(実装前に読むこと)

### A. Now Playing パネルの簡素化(ユーザー要望1)

`shell_content_now_playing`(tui.odin)から ROM/状態/fps/音量/スロットの詳細行を削除する。
タイトル行(`"BubiBoyLite v%s — Now Playing"`)は他画面(設定メニュー等)との一貫性のため
残す。メッセージログ部分はそのまま維持(ステータス行は直近1件のメッセージしか出せない
ため、履歴表示として引き続き価値がある)。

### B. ヒント文言の修正(バグ修正)

Now Playing ビュー用 `hint` を実態に合わせる。生ホットキーは存在しないため空文字にする
(空文字なら `tui_render_shell` がヒント自体を表示しない分岐が効く)。設定ビュー側の
ヒント(`"↑↓ 選択  ←→ 値を変更  Esc 戻る"`)はキー操作そのものなので変更不要。

### C. 最下部固定バグの調査・修正(仮説の検証)

1. alt screen 進入直後にスクロール領域リセット `\x1b[r`(パラメータなしの DECSTBM =
   全画面をスクロール範囲にする)を送出する。`ALT_SCREEN_ENTER` 定数自体に
   `"\x1b[?1049h\x1b[r"` として組み込み、送出箇所全てで漏れなく適用されるようにする
2. pty 検証で仮説を再現・検証する: 実バイナリ起動前に意図的に制限されたスクロール領域
   (`printf '\x1b[1;29r'` 相当)を pty に送ってから `./bbl` を起動し、旧コード
   (リセットなし)では区切り線/ステータス/入力行が29行目付近で止まること、修正後の
   コード(リセットあり)では実端末の最終行まで正しく描画されることを確認する
3. 上記で再現しない場合(=仮説が外れた場合)でも、DECSTBM リセットの追加自体は
   一般的なTUI作法として正しい防御策なので実施する

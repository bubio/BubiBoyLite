# BubiBoyLite 開発計画書（マスター）

BubiBoyLite（実行ファイル名 `bbl`）の開発計画のマスタードキュメント。
仕様の唯一の源は [BluePrint.md](BluePrint.md)。本計画書と矛盾があれば BluePrint が勝つ。

**実装者への最初の指示**: 作業を始める前に、このファイルの「進捗ダッシュボード」を見て現在のフェーズを特定し、
そのフェーズの詳細ファイル（`phases/phase-NN-*.md`）と [architecture.md](architecture.md) だけを読めばよい。
全フェーズファイルを読む必要はない。

## フェーズ依存グラフ

```
0 骨格 ──▶ 1 CPU ──▶ 2 タイマー/割り込み ──▶ 3 PPU ──▶ 4 MBC ──▶ 5 APU ──▶ 6 GBC ──▶ 7 ステート
                                                              │
                                                              ├──▶ 8 フロントエンド（5 完了後ならいつでも）
                                                              └──▶ 9 TUI（5 完了後ならいつでも、8 と並行可）
0 の最小 CI ────────────────────────────────────────────────────▶ 10 CI/CD 拡張 ──▶ 11 リリース
```

- 基本は番号順に進める。8 と 9 は 5 完了後なら 6/7 と並行できる。
- フェーズ 2 を PPU より先に置く理由: タイマー/割り込みは画面なしでテスト ROM 検証でき、
  PPU の STAT 割り込みがこの基盤に依存するため。
- 最小 CI はフェーズ 0 で作り、フェーズ 10 で全プラットフォームに拡張する。

## 進捗更新のルール（実装者は必ず守ること）

1. **タスク完了の条件**: タスクに書かれた「検証方法」のコマンドを実際に実行して成功を確認したときだけ、
   フェーズファイルのチェックボックスを `- [x]` にできる。**検証未実行のチェック付与は禁止**。
2. **検証ログ**: チェックを付けたら、フェーズファイル末尾の「検証ログ」に 1 行追記する。
   形式: `YYYY-MM-DD T2-3 完了: odin test 全パス, mooneye timer 8/8`
3. **PLAN.md の更新はフェーズ状態遷移時のみ**:
   - フェーズ最初のタスクに着手 → 状態を 🟡 に
   - フェーズのマイルストーン検証コマンドが成功 → 🟢 + 完了日を記入
   - 「タスク n/m」列は、フェーズファイル内のタスク見出し直下のチェックボックス数と常に一致させる
4. **ブロック時**: 状態を 🔴 にし、フェーズファイルに「ブロック理由」「試したこと」を記録してから中断する。
   次のセッションが文脈を回復できるようにするため。
5. **コミット規約**: 1 タスク = 1 コミットを目安とし、メッセージ先頭にタスク ID を付ける
   （例: `T2-3: implement TIMA falling-edge detection`）。ドキュメントの進捗更新は実装と同一コミットに含める。
6. **DoD の事後書き換え禁止**: 完了条件を変更する必要が生じたら、変更理由を検証ログに残した上で変更する。
7. **実際のユーザー名をコード・ドキュメント・コミットに残さない**。パスは `~` か環境変数で表記する（BluePrint の注意事項）。

## タスク記述規約（計画書メンテナ向け）

各フェーズファイルのタスクは以下のテンプレートに従う。**「参照」「完了条件」「検証方法」「落とし穴」は必須欄**。

```markdown
### T<フェーズ番号>-<連番>: タイトル

- [ ] 完了

**目的**: 1〜2 文。
**作るもの**: 作成/変更するファイル名を明示 + 実装ポイント。レジスタアドレス・ビット定義はここに直書きする
（references.md への参照だけで済ませない。タスクは自己完結が原則）。
**参照**: BubiBoy の対応ファイル（`~` パス表記）+ Pan Docs の該当章名。
**完了条件 (DoD)**: コマンド + 期待出力で書く。「正しく動くこと」のような主観表現は禁止。
**検証方法**: 実行するシェルコマンド。
**落とし穴**: テスト ROM が検出する既知のハマりどころを先回りして書く。
**依存**: 先行タスク ID。
```

粒度の目安: 1 タスクは 1 セッションで完結し、単体で検証できること（生成コード 100〜500 行程度）。

BubiBoy（`~/dev/_Emu/BubiBoy`、F# 製の前身）のコードは**アルゴリズムの参照元として移植してよい**。
ただし F# のイディオム（判別共用体、イミュータブル更新など）は Odin の tagged union / 構造体 + 手続きに読み替えること。
